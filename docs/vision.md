# プロジェクトビジョン

> 最終更新: 2026-02-14 | 根拠: [docs/discussion_20260214.md](discussion_20260214.md)

## このプロジェクトは何か

PPTXを題材にした**「AIと人間が構造的な創作物を協働で作るための仕組み」の実験**。

PowerPoint自動生成ツールではない。本当に解いている問題は:
- 構造的な創作物（PPTX、動画、文書等）をAIと人間が協働で作るとき、何が必要か
- AIの特性（コンテキスト制約、確率的出力）を前提としたパイプラインはどう設計すべきか
- 自己改善サイクルを持つ制作システムは実現できるか

PPTXが題材として優れている理由: 構造がある、創造性がある、品質判断が必要、1人で完結しがち。

---

## 発見された5つの原則

この実験を通じて言語化された、メディア非依存の原則:

### 1. 意図と実装の分離
「何を伝えたいか」と「どう実装するか」を分けないと、AIは座標やフォントサイズに引きずられる。Recipe層はこの発見の実装。

### 2. AIのコンテキストは有限のワーキングメモリ
全知識を一度に読み込むとコンテキストが枯渇する。Skillsアーキテクチャ（必要な時に必要な知識だけ読む）はこの制約への解。

### 3. 暗黙の制作段階を明示化する
人間が無意識に踏んでいる段階（構成→意図→レイアウト→調整）を、Tier 1→Recipe→Tier 2→Assembly→Reviewとして明示化。AIが各段階に参加できるようになる。

### 4. 分業には共通言語が必要
映像なら絵コンテ、建築なら設計図。AIとの分業ではJSON形式のRecipeとoutlineが共通言語。テンプレートの二重定義（profile.json + design_guide.md）も同じ原理。

### 5. 早い段階でレビューする
一括生成→最終確認ではなく、1単位ずつ対話。手戻りコストは段階ごとに指数関数的に増える。

---

## 現在のアーキテクチャ

```
Tier 1（アウトライン）→ Recipe（設計意図）→ Tier 2（テンプレート固有JSON）→ PPTX結合
     構成・流れ         何をどう見せるか      座標・色・フォントサイズ       最終出力
```

- **3層分離**: 意図→設計→実装を明確に分離
- **Skills**: AI知識のオンデマンド読み込み（共通 + テンプレート固有）
- **テンプレート二重定義**: profile.json（機械可読）+ design_guide.md（AI可読）
- **対話ベースのレビュー**: 1枚ずつ確認する段階的ワークフロー
- **全ファイルgit管理**: PPTX・サムネイル・Tier 2 JSON・画像を含め全てgit追跡

## 目指す姿 — 4つの発展方向

### A. 自己改善サイクル
生成→評価→修正→学習のループを自動化する。

- **形式的品質**（情報密度、配置バランス等）はルール化してAI自律で回す
- **意味的品質**（メッセージ、ストーリー）は人間が判断する
- 評価結果の構造化、修正戦略マップ、停止条件の定義が必要

### B. 模倣と守破離
参照作品ライブラリを構築し、手本から始めて独自に発展させるフロー。

- 良い作品からRecipeを逆算して参照ライブラリに登録
- Recipe生成時に類似の参照を提示（抽象原則ではなく具体的手本）
- 自己改善サイクルと接続: 良い成果物が次の参照になる（使うほど目が肥える）

### C. オープンコラボレーション
GitHubを中央ハブとして、任意の人間・AIが参加できる構造。

- JSON Schemaによる入出力契約
- エージェント非依存のワークフロー定義（WORKFLOW.md）
- GitHub Issues/PRによる作業管理・レビュー
- GitHub Actionsによる自動検証・組み立て

### D. メディア非依存化
同じ3層構造をPPTX以外（動画、文書等）にも適用。

- Tier 1/Recipe/Tier 2の抽象インターフェース定義
- メディア固有の部分（engine, template, assembly）をプラグイン化
- ただし現段階では過剰設計。PPTXで完成させてから抽象化する

## 開発フェーズ

詳細は [docs/ROADMAP.md](ROADMAP.md) を参照。

| Phase | 名称 | 目的 |
|---|---|---|
| 0 | 知識の外部化 | 暗黙知をドキュメント化。別のAIが「なぜ」を理解できる状態 |
| 1 | 契約の定義 | JSON Schema + WORKFLOW.md。誰でも「何を出力すべきか」分かる状態 |
| 2 | 自動化基盤 | GitHub Actions。PRを出せば自動で検証される状態 |
| 3 | 参照ライブラリ | 模倣の基盤。「手本を見て作る」が可能な状態 |
| 4 | 自己改善ループ | 形式的品質がAI自律で改善される状態 |
